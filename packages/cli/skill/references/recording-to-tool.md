# 从录制到代码落地

这份参考页回答的是：

**拿到一份浏览器录制结果之后，怎样把它整理成一个或一组工具，并最终落到技能代码里。**

它不只讲“怎么分析录制”，还讲：

- 录制结束后先做什么
- 怎么判断该做一个工具还是多个工具
- 最后需要改哪些文件

## 什么时候看这份参考

适合在下面这些阶段看：

- 刚完成一轮 `cli-skill browser record`
- 已经拿到了 `summaryPath`
- 准备从录制结果提炼工具
- 想把录制内容真正落到 `src/tools/*` 和 `src/skill/*`

如果你现在还没开始录制，先回到：

- [主技能说明](../SKILL.md)

## 正确的默认顺序

浏览器场景默认按这个顺序做：

1. 启动 `cli-skill browser record`
2. 等待录制命令结束，并拿到最终返回 JSON
3. 先看 `stopReason`
4. 适合继续时，再从 `summaryPath` 开始读录制结果
5. 先写“用户做了什么”
6. 再判断适合沉淀成哪些工具
7. 再提炼参数、输出和成功判定
8. 最后才生成工具代码并补技能文档

## 第一步：先确认这次录制能不能继续用

命令退出后，先看返回 JSON：

- `summaryPath`
- `stopReason`

默认规则：

- `user_stop`
  - 这次录制是用户主动结束，可以继续分析
- `browser_closed`
  - 这次录制是中断，默认先判断内容是否足够
- `error`
  - 先修问题，不要直接继续生成工具

## 第二步：先看哪些文件

优先按这个顺序看：

1. `summary.json`
2. `timeline.jsonl`
3. `actions.jsonl`
4. `dom.jsonl`
5. `network.jsonl`

它们分别回答：

- `summary.json`
  - 录制是否完整结束
  - 最后停在哪个页面
  - 这次大概有多少动作、请求、DOM 快照
- `timeline.jsonl`
  - 整条流程的主线
  - AI 默认先读它
- `actions.jsonl`
  - 用户显式做了什么
  - `timeline.jsonl` 里的 `action`、`navigation`、`tab_switch` 事件通过 `actionId` 指到这里
- `dom.jsonl`
  - 页面在哪些点发生了结构变化
  - `timeline.jsonl` 里的 `dom_changed` 事件通过 `domSnapshotId` 指到这里
- `network.jsonl`
  - 后台真实完成了什么业务动作
  - `timeline.jsonl` 里的 `request_started`、`request_finished` 事件通过 `networkId` 指到这里

默认阅读顺序是：

1. 先看 `summary.json`
   - 判断录制是否完整、最后停在哪个页面
2. 再看 `timeline.jsonl`
   - 按时间理解整条流程
3. 只有在某个时间点需要更多上下文时，再按事件里的 id 去查对应明细
   - `actionId` -> `actions.jsonl`
   - `domSnapshotId` -> `dom.jsonl`
   - `networkId` -> `network.jsonl`

## 第三步：先写“用户做了什么”

先把流程翻译成自然语言，不要直接写定位器，也不要直接写代码。

至少写出：

- 从哪个页面开始
- 打开了什么入口
- 填了哪些字段
- 做了哪些选择
- 提交后进入了哪个详情页
- 后续又做了哪些动作

例如：

- 打开搜索页面
- 输入查询关键词
- 提交搜索
- 打开结果页里的第一条结果
- 在结果详情页继续执行后续动作

这一步的目的是确认：

**录制里真正的业务目标是什么。**

## 第四步：判断该做一个工具还是多个工具

不要默认“一次录制 = 一个工具”。

先判断这次录制更像下面哪一种：

- 一个完整、单一、可重复的业务动作
- 多个连续但可拆分的业务动作
- 一个探索过程，其中只有一部分值得沉淀

### 适合做成一个工具的情况

- 用户目标本来就是“一次完成”
- 中间结果没有独立复用价值
- 整体更像一个业务动作，而不是多个松散动作

例如：

- 查询关键词并打开第一条搜索结果
- 登录后台并创建一条内容

### 适合拆成多个工具的情况

- 一次录制里实际做了多个业务阶段
- 中间产物本身有复用价值
- 后续步骤依赖前一步的输出
- 有些步骤适合 DOM，有些步骤适合 API

例如：

- 先登录站点
- 再进入管理页面
- 再创建内容
- 最后发布内容

## 第五步：提炼参数、默认值和输出

### 参数

先把字段分成两类：

- **显式输入**
  - 用户真的填了、选了、改了
- **默认值**
  - 页面自动带上的，用户没有主动动

优先从这些地方找：

- `actions.jsonl`
  - 用户输入了什么
- `network.jsonl`
  - 请求体里最终提交了什么
- `dom.jsonl`
  - 表单里还有哪些字段存在

默认原则：

- 显式输入优先做成参数
- 默认值只有在业务上常改、且对结果重要时，才升级成参数

### 输出

一个工具的输出应该服务于下一步使用。

优先考虑返回：

- 新对象的 id
- 详情页 URL
- 关键业务状态
- 后续动作是否已触发

例如：

- `resultId`
- `resultUrl`
- `title`
- `status`
- `submitted`

### 成功判定

成功判定比定位器更重要。

优先级建议是：

1. 页面跳转到明确 URL
2. 页面上出现明确标题、按钮或状态
3. 关键接口返回成功
4. 详情页能读到新对象 id 或状态

例如：

- 提交成功后跳到 `/detail/<id>`
- 页面上出现“提交成功”或“发布成功”
- `POST /api/create` 返回成功
- `POST /api/publish` 返回成功

## 第六步：决定走 DOM 还是 API

录制完成后，不要默认继续走 DOM。

判断方式可以很简单：

- 如果页面结构清楚、动作主要是表单和按钮
  - 优先考虑 DOM
- 如果关键业务动作已经清楚对应一个稳定接口
  - 优先考虑 API
- 如果 UI 只是帮助理解参数，接口更稳
  - 录制用于理解，工具用 API
- 如果页面有很多默认状态，接口不容易独立复用
  - 先做 DOM

录制的作用不是强迫你走哪一种，而是帮助你做这个选择。

## 第七步：把结论落成代码

当你已经判断清楚“做什么工具、参数是什么、成功判定是什么”之后，默认落地顺序是：

1. 在 `src/tools/*` 新建或修改工具文件
2. 在 `src/index.ts` 接入工具
3. 在 `src/skill/SKILL.md` 补：
   - 技能目标
   - 适用场景
   - 典型流程
   - 工具输入输出和关键约束
4. 运行：

```bash
cli-skill build
```

5. 需要给 agent 用时，再运行：

```bash
cli-skill mount
```

## AI 最好先输出什么

在真正生成工具之前，AI 最好先输出一份结构化结论，而不是直接给代码。

推荐至少包含：

### 1. 用户行为描述

- 用户从哪里开始
- 做了哪些主要动作
- 最终停在哪个页面

### 2. 候选工具

- 这次录制适合沉淀成哪些工具
- 每个工具负责哪一段流程

### 3. 参数建议

- 哪些字段应作为输入参数
- 哪些字段可以继续沿用默认值

### 4. 输出建议

- 每个工具执行后应该返回什么

### 5. 成功判定

- 每个工具如何判断自己执行成功

### 6. 实现建议

- 更适合 DOM 还是 API
- 当前录制里还有哪些信息不足
