# 从录制到工具

这份参考页只解决一件事：

**当你已经拿到一份录制结果时，怎样把它翻译成工具设计。**

它不负责讲：

- 项目怎么创建
- 技能源码怎么分层
- build、mount、publish 的平台规则

那些内容留在：

- [主技能说明](../SKILL.md)
- [技能源码参考](skill-source.md)

## 什么时候看

适合在下面这些时刻看：

- 刚完成一轮 `cli-skill browser record`
- 已经拿到了 `summaryPath`
- 浏览器工具执行失败，准备先复盘这次运行记录
- 已经决定要把这段流程沉淀成工具，但还没开始写代码

如果你现在还没开始录制，先回到：

- [主技能说明](../SKILL.md)

## 默认顺序

拿到录制结果后，默认按这个顺序做：

1. 先确认这次录制值不值得继续分析
2. 先读 `summary.json`
3. 再读 `timeline.jsonl`
4. 先写出“用户做了什么”
5. 再判断该做一个工具还是多个工具
6. 再提炼参数、输出和成功判定
7. 最后才回到源码里落代码

如果这次记录不是手动录制，而是浏览器工具失败后留下的运行记录，也按这套顺序分析。

## 第一步：先确认这次记录能不能继续用

先看：

- `summary.json`

最重要的是两件事：

- 这次流程有没有录到关键步骤
- 最后停在了哪个页面

如果这是一次失败后的浏览器工具运行记录，再额外先回答：

- 失败之前最后一个成功动作是什么
- 失败更像：
  - 选择器问题
  - 页面时序问题
  - 登录态问题
  - 页面结构变化问题

## 第二步：先看哪些文件

优先按这个顺序看：

1. `summary.json`
2. `timeline.jsonl`
3. `actions.jsonl`
4. `dom.jsonl`
5. `network.jsonl`

它们的分工是：

- `summary.json`
  - 这次录制是不是完整
  - 最后停在哪个页面
  - 动作、请求、DOM 变化大概有多少
- `timeline.jsonl`
  - 主线
  - 默认先读它
- `actions.jsonl`
  - 用户显式做了什么
- `dom.jsonl`
  - 页面在哪些点发生了变化
- `network.jsonl`
  - 后台真实完成了什么业务动作

默认阅读方式是：

1. 先读 `summary.json`
2. 再读 `timeline.jsonl`
3. 只有某个时间点需要更多上下文时，再按 id 去查明细：
   - `actionId` -> `actions.jsonl`
   - `domSnapshotId` -> `dom.jsonl`
   - `networkId` -> `network.jsonl`

## 第三步：多页面流程怎么读

如果录制过程中打开了新页面，不要把整条时间线硬读成“一个页面的连续动作”。

默认规则：

- 同一个录制里可能同时存在多个 `pageId`
- 旧页面后面仍然可能继续产生 DOM 变化或请求
- `actionId`、`networkId`、`domSnapshotId` 用来定位具体事件
- `pageId` 用来判断这条事件到底属于哪个页面

所以当你看到：

- `tab_switch`
- 新页面跳转
- 明显进入一个新的 URL

后面的动作、请求和 DOM 变化，要结合 `pageId` 一起读。

## 第四步：先写“用户做了什么”

这里不要直接写代码，也不要直接开始猜 DOM。

默认先输出一个短结构：

```md
任务目标
- <这次流程真正想完成什么>

起点
- <从哪个页面或入口开始>

关键步骤
1. <进入哪个入口>
2. <填写了什么字段>
3. <做了什么选择>
4. <触发了什么提交或跳转>

结果
- <最后到了哪个页面、创建了什么对象、拿到了什么结果>

待确认
- <哪些地方还需要继续查 actions / network / dom>
```

要求：

- 每一步只写用户能理解的业务动作
- 不要写定位器
- 不要写代码
- 不要把所有细节一股脑铺开
- 优先让人一眼看出“这次流程到底完成了什么”

如果流程很长，也先压成 4 到 8 个关键步骤，再在后面按需展开。

## 第五步：判断该做一个工具还是多个工具

不要默认“一次录制 = 一个工具”。

先判断这次流程更像：

- 一个完整、单一、可重复的业务动作
- 多个连续但可拆分的业务动作
- 一个探索过程，其中只有一部分值得沉淀

更适合做成一个工具的情况：

- 用户目标本来就是一次完成
- 中间结果没有独立复用价值
- 整体更像一个业务动作，而不是多个松散动作

更适合拆成多个工具的情况：

- 一次流程里实际做了多个业务阶段
- 中间产物本身有复用价值
- 后续步骤依赖前一步的输出
- 有些步骤更适合 DOM，有些步骤更适合 API

## 第六步：提炼参数、输出和成功判定

### 参数

优先从这里找：

- `actions.jsonl`
  - 用户真的填了、选了、改了什么
- `network.jsonl`
  - 最终请求里真正提交了什么
- `dom.jsonl`
  - 页面上还存在什么表单字段和状态

默认原则：

- 用户显式输入优先做成参数
- 页面自动带上的默认值，不要轻易升级成参数

### 输出

一个工具的输出应该服务于下一步使用。

优先考虑：

- 新对象 id
- 详情页 URL
- 关键业务状态
- 是否已提交、已发布、已触发

### 成功判定

成功判定优先级建议是：

1. 页面跳转到明确 URL
2. 页面上出现明确标题、按钮或状态
3. 关键接口返回成功
4. 详情页能读到新对象 id 或状态

## 第七步：决定走 DOM 还是 API

录制的作用不是强迫你继续走 DOM，而是帮助你判断哪条路更稳。

默认判断：

- 页面结构清楚，动作主要是表单和按钮
  - 优先考虑 DOM
- 关键业务动作已经能对应到稳定接口
  - 优先考虑 API
- UI 只是帮助理解参数，接口更稳
  - 录制用于理解，工具用 API

## 第八步：什么时候回到源码

当下面这些问题已经回答清楚时，就不要继续停留在录制分析里了：

- 这次流程到底想完成什么
- 该做一个工具还是多个工具
- 参数是什么
- 输出是什么
- 成功判定是什么

这时回到：

- [技能源码参考](skill-source.md)

开始真正改 `src/tools/*`、`src/index.ts` 和 `src/skill/*`。
