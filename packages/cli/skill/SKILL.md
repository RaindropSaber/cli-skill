# cli-skill

这个技能不是一份平台说明书，而是一份默认工作流。

它解决的是同一类任务：

- 把一个想法整理成可复用的技能项目
- 把一段浏览器流程沉淀成一个或多个工具
- 维护、安装、挂载、发布已有技能

当用户想要的是“做出一个技能”或“把流程沉淀成技能能力”时，优先使用这个技能。

## 先理解这件事

`cli-skill` 的目标不是让 agent 背命令，而是让 agent 按正确顺序做事。

默认顺序应该是：

1. 先判断用户到底要新建、修改、安装，还是把一段流程沉淀成工具
2. 如果是浏览器流程，先录制真实操作
3. 先解释流程，再决定做一个工具还是多个工具
4. 再落代码、补文档、构建和挂载

如果顺序反了，后面通常会出现这些问题：

- 直接猜 DOM，工具很快失效
- 还没想清业务目标，就把一次录制硬写成一个工具
- 代码改完了，但 agent 文档没补，技能最后不好用

## 默认原则

### 先建项目，再补能力

如果用户要的是一个新技能，先创建项目骨架，再写工具和文档。不要跳过项目结构，直接散着写脚本。

### 浏览器场景先录制

如果用户要的是浏览器里的真实流程，默认不要先猜实现。

正确做法是：

1. agent 自己在目标技能目录里启动 `cli-skill browser record`
2. 直接打开录制浏览器，让用户在浏览器里完成真实操作
3. 持续等待录制结束，不要把命令甩给用户自己执行
4. 录制结束后先分析结果，再决定代码怎么写

### 先写流程，再写代码

拿到录制结果后，先回答：

- 这次流程真正想完成什么
- 用户实际做了哪些关键步骤
- 哪些步骤值得沉淀
- 该做一个工具还是多个工具
- 输入、输出和成功判定分别是什么

这里的“用户做了什么”不要写成长段散文，默认先产出：

- `任务目标`
- `起点`
- `关键步骤`
- `结果`
- `待确认`

### 最后一定补技能文档

代码完成后，必须回到 `src/skill/*`，把“这个技能以后怎么被用”补清楚。

这里不是简单补一句概述，而是至少补完下面这些信息：

- 这个技能真正解决什么问题
- 用户在什么场景下会触发它
- 一条最常见的默认流程是什么
- 每个关键工具分别负责哪一段
- 工具的输入、输出和成功结果是什么
- 这个技能依赖哪些前置条件、环境或登录态
- 哪些情况不适合继续用这个技能硬做

默认检查顺序：

1. 先补 `适用场景`
   - 让后续 agent 知道什么时候该用这个技能
2. 再补 `触发示例`
   - 用真实用户会说的话触发，不要写平台术语
3. 再补 `推荐流程`
   - 把最常见的业务路径压成 4 到 8 步
4. 再补 `结果说明`
   - 让下游知道哪些输出最值得继续使用
5. 再补 `配置约定`
   - 说明需要哪些 key，推荐放全局还是项目目录
6. 最后补 `使用边界` 和 `维护提示`
   - 让后续 agent 知道什么时候该停下来确认，什么时候改完代码还要同步改文档

如果这个技能包含浏览器工具，额外补清楚：

- 失败时是否建议开启运行记录
- 失败后默认先读哪些文件
- 如果流程里会打开多个页面，后续记录需要结合 `pageId` 一起读

不要只停在模板默认的最小占位内容。

## 默认流程

### 情况一：新建技能

当用户要新建一个技能时：

1. 用 `cli-skill create` 创建项目
2. 安装依赖
3. 明确技能目标
4. 实现工具
5. 补 `src/skill/*`
6. 执行 `cli-skill build`
7. 如果要给 agent 用，再执行 `cli-skill mount`

### 情况二：浏览器流程沉淀

当用户要把一段浏览器操作沉淀成工具时：

1. agent 自己启动 `cli-skill browser record`
2. 等用户完成真实操作
3. 持续等待录制命令结束
4. 先读 `summary.json`
5. 再读 `timeline.jsonl`
6. 需要时按 `actionId`、`networkId`、`domSnapshotId` 查明细
7. 先写出业务流程
8. 再决定工具拆分、参数和成功判定
9. 最后才改 `src/tools/*`、`src/index.ts`、`src/skill/*`

### 情况三：维护已有技能

当用户要改一个已有技能时：

1. 先读 `src/index.ts`、`src/tools/*`、`src/skill/*`
2. 判断这次是在改工具、改文档，还是改项目工作流
3. 修改后执行 `cli-skill build`
4. 如果需要给 agent 用，再执行 `cli-skill mount`

### 情况四：安装或发布技能

安装和发布都先按“包”来理解，不按“技能名搜索”来理解。

- 安装已发布技能时，直接使用包名
- 如果要给 agent 用，安装后还要再执行一次 `mount`
- 如果全局或项目配置了其他 registry，`install` / `publish` 都应该尊重该配置

- 技能源码该怎么落地：

- 细碎命令说明

堆回这份主文档里。

## 关键约束

- `cli-skill browser record` 是长任务，agent 启动后必须等它真正结束
- `build` 负责把 `src/skill/*` 渲染成根目录 `skill/*`
- `mount` 负责接通 bin，并把技能挂到 agent 目录
- `install` 只负责安装包、注册本机信息、接通技能 bin；如果要给 agent 用，还要再 `mount`
- 配置按文件层级覆盖：
  - 全局：`~/.cli-skill-config.json`
  - 本地：当前目录及父目录里的 `.cli-skill-config.json`
- 平台不会再偷偷替技能做配置隔离；如果要隔离，由技能自己定义 key
